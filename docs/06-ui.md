# UI & Web Server

The web server is a `Bun.serve()` process that serves the operator UI, agent-created pages, and agent-defined API routes. Everything is files on disk, hot-reloaded via Bun's native HMR.

## Architecture

The UI follows the shadcn model — components are files in the project, not an installed package. They can be added via CLI tooling and modified directly. The agent has the same access: it can create, edit, or delete any UI file using its core tools.

### Bun's Native Frontend Pipeline

Bun handles the full frontend build pipeline with zero external tooling:

- **HTML as route handlers** — `import page from "./page.html"` registers it as a route in `Bun.serve()`. Bun scans the HTML for `<script>` and `<link>` tags, bundles everything, rewrites paths with content hashes.
- **JSX/TSX** — native transpilation and bundling. No Babel, no esbuild. A `.tsx` file referenced from HTML is bundled automatically.
- **CSS** — bundling, nesting, CSS Modules (`.module.css`), modern syntax lowering, vendor prefixing. All built-in.
- **Tailwind** — via `bun-plugin-tailwind`. Single plugin, configured in `bunfig.toml`.
- **React Fast Refresh** — automatic for React components during HMR. Component state preserved across edits, no manual `import.meta.hot` calls needed.
- **Production builds** — `bun build` produces minified, content-hashed output. Zero-config.

### How Agent Edits Become Live

1. Agent writes/edits a `.tsx`, `.html`, or `.css` file
2. Bun's HMR detects the change
3. Browser updates without full reload (React Fast Refresh for components)

No build step, no watcher config, no bundler setup. Agent writes file → UI updates.

### File-Based Routing

`Bun.FileSystemRouter` provides Next.js-style file routing out of the box:

```ts
const router = new Bun.FileSystemRouter({
  style: "nextjs",
  dir: "./pages",
});
```

Supports dynamic params (`[slug].tsx`), catch-alls (`[[...path]].tsx`), and index routes. When the agent creates a new page file, calling `router.reload()` rescans the directory and the route is live. Paired with a file watcher, this gives us automatic route discovery with zero custom code.

## What the Agent Can Do

### Pages
Create new pages by writing an HTML file that imports a `.tsx` entrypoint. Drop it in the pages directory, the framework registers the route, Bun bundles and serves it.

```html
<!-- pages/dashboard.html -->
<html>
  <head>
    <link rel="stylesheet" href="./dashboard.css" />
  </head>
  <body>
    <div id="root"></div>
    <script type="module" src="./dashboard.tsx"></script>
  </body>
</html>
```

### Components
Add components to existing views. Write a `.tsx` file, import it from an existing page. Bun bundles the dependency and HMR picks up the change.

### API Routes
Write route handler files that the framework registers into `Bun.serve()`:

```ts
// routes/webhook-github.ts
export default {
  POST(req) {
    const payload = await req.json();
    // trigger work via event bus
    return new Response("ok");
  }
};
```

This enables webhook receivers, form handlers, and custom integrations. Note: these are not transports. Transports are bidirectional conversation channels. A webhook handler is one-way inbound — it receives a payload and triggers work via the event bus, with no ongoing conversation back to the source. Transports may use webhooks internally, but an agent writing a standalone webhook is just wiring up a trigger.

## Default Views

The framework ships with a minimal set of operator views:

- **Thread overview** — all threads, their status, transport, last active time
- **Conversation view** — read and follow a thread's message history in real-time (streaming responses, tool calls, etc.)
- **Event bus** — upcoming scheduled/cron events, recent event history
- **Skills browser** — installed skills, their descriptions

These are all shadcn-style files in the project. The agent can modify or extend them.

## Operator as Transport

The UI itself is a transport. Operators can:

- **Create new threads** — start a conversation with the agent directly in the UI
- **Send messages** to existing threads (prompt, steering, follow-up)
- **Stop** a running agent thread
- **Trigger events** manually

This means the UI transport works like any other transport — it logs to channel history, syncs on invocation, delivers responses. The "channel" is just the browser session.

## Real-Time

`Bun.serve()` has native WebSocket support. The UI maintains a WebSocket connection for:

- Streaming agent responses as they generate
- Tool execution progress
- New messages arriving in threads
- Event bus activity

```ts
Bun.serve({
  routes: { ... },
  websocket: {
    open(ws) { /* subscribe to agent events */ },
    message(ws, msg) { /* handle operator input */ },
    close(ws) { /* cleanup */ },
  },
});
```

## Tech Stack

- **Runtime**: `Bun.serve()` with HTML imports
- **Framework**: React (client-rendered SPA)
- **Styling**: Tailwind CSS via `bun-plugin-tailwind`
- **Components**: shadcn-style, files in project
- **Real-time**: Native WebSocket
- **Build**: Bun's built-in bundler (dev: HMR, prod: `bun build`)

No Vite, no webpack, no Next.js, no Express.

## Open Questions

- **Auth**: basic auth? OAuth? Behind a VPN? Personal agent on a server needs some protection — mechanism TBD.
- **SSR**: Bun supports SSR via `renderToReadableStream` from `react-dom/server`. Not automatic, but a thin helper could make route handlers return JSX directly. Worth providing this as a convenience for simpler agent-created pages that don't need a full client-side bundle.
- **Agent route limits**: should there be any restrictions on what routes the agent can create? Or full freedom?
